### ✅ What Works Well

1. **Three-layer hierarchy implemented correctly:**
   - Layer 1 (plan panel widget): Shows goal + step progress with `◆` prefix
   - Layer 2 (chat inline): Shows detailed step/substep progress with `✓/⠋/○` icons
   - Layer 3 (peek overlay): `Ctrl+Q` shortcut registered

2. **Icon system is consistent:**
   - `✓` for completed
   - `⠋⠙⠹...` for active (10 frames, 80ms cycle)
   - `○` for pending
   - Matches SPEC-UI.md exactly

3. **Progress dots row works:**
   - `●○○ 1/3` format with blinking active dot
   - Properly updates as steps complete

4. **Substep detail under active step:**
   - Shows `→ Reading auth.ts` ephemeral detail during tool execution
   - Clears on tool completion
   - Collapses completed substeps

5. **Duration formatting:**
   - `45s`, `2m 13s` format
   - Only shown on completed steps
   - Correct per SPEC-UI.md

### ⚠️ UI/UX Gaps

#### 3.1 Completed Substeps Collapse vs. Erase (VISION.md Violation)

**VISION.md says:**
> "substeps collapse into their parent step when that step completes. This keeps the feed clean while providing real-time granularity."

**Current behavior:** In `activity-feed.ts`, when a step completes, the substeps remain visible under the completed step. The VISION.md says they should collapse (be hidden under the parent, not erased).

**Status:** Actually, re-reading the code, completed substeps DO remain visible in the rendered output. The VISION.md says "collapse not erase" — meaning they should still be accessible but hidden by default. The current implementation shows them inline. This is acceptable but not matching the "collapse" intent (which implies a toggleable collapsed state).

**Grade:** ⚠️ Partial — substeps don't collapse, they stay visible. Not a bug, but not matching VISION.md intent.

#### 3.2 Smart Goal Summarization (VISION.md Violation)

**VISION.md says:**
> "Never use the user's raw input as the goal. The model must comprehend and summarize."

**Current behavior:** `plan-panel.ts:summarizeGoal()` does basic regex stripping of URLs and code blocks, but doesn't use LLM summarization. The goal is often the raw prompt text.

```typescript
// plan-panel.ts:380
export function summarizeGoal(goal: string): string {
    let cleaned = goal.replace(/https?:\/\/[^\s]+/g, '');
    cleaned = goal.replace(/```[\s\S]*?```/g, '');
    // ... basic regex only
}
```

**Fix:** Use LLM to summarize or accept that regex cleaning is sufficient for MVP.

#### 3.3 Peek Overlay is 90% Built, Disconnected

**VISION.md says:**
> "The peek is essential for trust. Without the peek, the user sees only orchestration-level steps."

**Current behavior:** `peek-overlay.ts` has the full implementation, but `subagent-runner.ts` never calls `updatePeek()` or `updatePeekFeed()` with live data.

```typescript
// subagent-runner.ts: session.subscribe()
// NO calls to updatePeek() or updatePeekFeed()
```

**Impact:** `Ctrl+Q` opens an empty overlay with no live content. The `_peekLines` array is never populated.

**Fix:** Add `updatePeek(event.delta)` calls in the `message_update` handler and `updatePeekFeed(feed)` calls after feed updates.

#### 3.4 Plan Panel Budget Management

**SPEC-UI.md says:**
> "Plan panel capped at 9 lines. Overflow trims oldest completed steps from top."

**Current behavior:** `plan-panel.ts` has a `BUDGET = 9` constant and trims old rows. ✅ Correct.

**However:** When trimming, the code shows `… +N more` for hidden steps, but the active step and its substeps always get priority. This is correct behavior.

#### 3.5 Tool Detail Rendering

**SPEC-UI.md says:**
> "Tool detail: `        <spinner> Running: <command>` under active substep only"

**Current behavior:** Tool detail is shown at 8-space indent with spinner. ✅ Correct.

**But:** The ephemeral tool detail line is rendered as `→` in some places and `⠋` in others. Need to verify consistency.

## 4. CODE QUALITY EVALUATION

### ✅ Architecture Strengths

1. **Modular design:** Clean separation into 11 files, each with single responsibility
2. **Type safety:** TypeScript interfaces in `types.ts` for all major data structures
3. **Timer safety:** `plan-panel.ts` uses global registry for timer self-check on hot reload
4. **Scope isolation:** `subagent-runner.ts` writes `.pi/scope.json` for `scope-guard.ts` enforcement
5. **Event-driven:** Subagent events properly wired through `session.subscribe()`
6. **No circular dependencies:** Dependency graph is acyclic

### ⚠️ Critical Issues

#### 4.1 State Machine Mutates Directly (Race Condition)

**File:** `activity-feed.ts`

**Problem:** Multiple event handlers mutate `feed` state directly without synchronization:

```typescript
// subagent-runner.ts: ~line 183
session.subscribe((event) => {
  if (event.type === "message_update") {
    output += event.assistantMessageEvent.delta;
    feed = parseTextForFeed(feed, event.assistantMessageEvent.delta);  // reassign
    updatePeekFeed(feed);
    // ...
  }
  if (event.type === "tool_execution_start") {
    feed = setToolDetail(feed, substepLabel);  // reassign
    // ...
  }
  if (event.type === "tool_execution_update") {
    // modifies feed.steps[feed.currentStep].substeps[last] directly!
    const lastSub = activeStep.substeps[activeStep.substeps.length - 1];
    lastSub.outputPreview = preview;  // DIRECT MUTATION
  }
  if (event.type === "tool_execution_end") {
    feed = clearToolDetail(feed);  // reassign
    feed = completeLastSubstep(feed, outputPreview);  // reassign
  }
});
```

**Race condition scenario:**
1. `message_update` fires at T=0ms → calls `parseTextForFeed()`
2. `tool_execution_update` fires at T=10ms → mutates `lastSub.outputPreview` directly
3. `message_update` fires again at T=20ms → uses stale `feed` reference

The problem is that `parseTextForFeed` returns a new object, but `tool_execution_update` mutates the OLD object's nested array. The `feed` variable is reassigned, but the event handlers are closures over the mutable state.

**Impact:** UI inconsistencies — substep output preview may appear on wrong substeps, or not appear at all. Hard to reproduce but will happen under load.

**Fix:** Use immutable reducer pattern for ALL state changes. Never mutate nested properties. The `parseTextForFeed`, `setToolDetail`, `clearToolDetail`, `completeLastSubstep` functions already return new state — the issue is the `tool_execution_update` handler which mutates directly.

**Also:** The `renderTimer` in `subagent-runner.ts` (80ms interval) calls `renderActivityFeed()` which reads `feed` state. If `feed` is being mutated while being read, renderer sees inconsistent state.

#### 4.2 `peek-overlay.ts` Interface Redeclaration

**File:** `peek-overlay.ts:23-30`

```typescript
interface Component {
    render(width: number): string[];
    handleInput?(data: string): void;
    invalidate(): void;
}
```

**Problem:** Re-declares pi-tui interfaces locally instead of importing. If the real interface changes (adds required method), code compiles but fails at runtime.

**Fix:** Import from `@earendil-works/pi-tui` directly. The `tsconfig.json` already maps it.

#### 4.3 System Prompt Injection is Fragile

**File:** `index.ts`

```typescript
const delegationInstructions = `## Orchestrator Mode...`;
return { systemPrompt: cleanedPrompt + delegationInstructions };
```

**Problem:** If `cleanedPrompt` is `undefined`, you get `"undefined[string content]"`. If it already contains `## Orchestrator Mode`, you get duplicate sections.

**Fix:** Use the existing guard pattern but ensure `basePrompt` is never undefined:
```typescript
const basePrompt = event.systemPrompt ?? "";
const marker = "## Orchestrator Mode";
if (basePrompt.includes(marker)) return { systemPrompt: basePrompt };
return { systemPrompt: basePrompt + "\n\n" + delegationInstructions };
```

#### 4.4 Subagent Runner Error Handling

**File:** `subagent-runner.ts:333`

```typescript
try {
  await session.prompt(task);
} catch (error) {
  // ... sets error state
} finally {
  // ... cleanup
}
```

**Problem:** If `session.prompt()` throws, the error is caught and `output` is set to `[error] ...`. But the `result` returned to `delegate-tool.ts` has `output` starting with `[error]`, which triggers `isError` check. This is correct.

**However:** The `toolCallTrail` is not built if the session errors early. The `result.toolCallTrail` will be empty. Minor issue.

#### 4.5 Scope Caching Logic

**File:** `delegate-tool.ts:37`

```typescript
let _cachedScope: Scope | null = null;
```

**Problem:** Scope is cached globally (module-level). If orchestrator calls `delegate("coder", ...)` twice in succession, the second coder gets the first coder's scope. Also, if `delegate("coder")` is called without explicit scope, it uses cached scope from previous scout — but what if the scout was from a different task?

**Fix:** Tie scope to delegation context, not global module state. Or clear scope after each coder delegation.

#### 4.6 Timer Leak in `delegate-tool.ts` Render

**File:** `delegate-tool.ts:95-105`

```typescript
if (isPartial && !state.interval) {
    state.interval = setInterval(() => {
        getSpinnerIndex(); // tick shared spinner
        context.invalidate();
    }, 80);
}
if (!isPartial && state.interval) {
    clearInterval(state.interval);
    state.interval = undefined;
}
```

**Problem:** The `renderResult` function sets up a timer that ticks every 80ms. If `renderResult` is called multiple times (e.g., due to re-renders), the timer might leak. The check `!state.interval` guards against this, but `state` is per-component-instance and managed by the TUI framework. Should be safe, but worth verifying.

## 5. WORKFLOW EVALUATION (Execution Trace)

The user provided a detailed execution trace of `delegate("scout", "investigate auth bug")`. Let me evaluate each step:

### ✅ Workflow Strengths

1. **Correct delegation flow:** Orchestrator → `delegate-tool.ts` → `runSubagent()` → isolated subagent session
2. **Proper tool restriction:** Subagent only gets `["read", "bash"]` — no `edit`, `write`, `plan`, `delegate`
3. **No context files:** `noContextFiles: true` means subagent doesn't inherit AGENTS.md or project context
4. **System prompt override:** Specialist's system prompt replaces the default, not appended
5. **Event streaming:** `message_update`, `tool_execution_start`, `tool_execution_end` all properly wired
6. **Output parsing:** `## Goal`, `## Steps`, `## Findings`, `## Scope`, `## Audit` all extracted from subagent output
7. **Metadata prepending:** `[Execution: elapsed=45.2s, turns=5]`, `[Tool Calls (3): ...]`, `[Findings: ...]`, `[Audit: ...]` all prepended correctly
8. **Plan panel update:** `startDelegationStep()` → `completePlanStep()` lifecycle correctly managed

### ⚠️ Workflow Gaps

#### 5.1 No `## Scope` Extraction from Scout Output

The trace shows scout outputs `## Scope` section with `filesToModify`, `filesToCreate`, etc. But `delegate-tool.ts` only extracts `## Findings` and `## Audit` — it does NOT extract `## Scope` to populate the coder's scope parameter.

**Current behavior:** Coder scope must be explicitly passed by the orchestrator in the `delegate()` call. The scout's `## Scope` output is visible in the text but not automatically extracted for the next coder call.

**Fix:** Add `extractScopeFromOutput()` function in `delegate-tool.ts` that parses the `## Scope` block and stores it in `_cachedScope` for automatic use by the next coder delegation.

#### 5.2 Subagent Cannot Signal "I Need More Tools"

If a scout discovers it needs to write a file (e.g., to create a test script), it cannot — it's restricted to `["read", "bash"]`. The orchestrator would need to spawn a coder instead.

**This is by design** — the scout is read-only. But the workflow doesn't explicitly handle the case where the scout needs to escalate to a coder.

#### 5.3 No Retry Logic for Failed Delegations

If a subagent errors (e.g., network timeout, model error), the orchestrator sees `[error]` and must decide to retry. There's no automatic retry with backoff.

**This is by design** — the orchestrator decides. But the trace doesn't show any retry guidance in the system prompt.
