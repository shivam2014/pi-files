# Biggest Positive

The strongest architectural decision is:

```text
Orchestrator
    ↓
Scout
    ↓
Coder
    ↓
Reviewer
```

with tool-level gating.

This is exactly the sort of mechanism that survives prompt drift.

Prompt rules decay.

Tool restrictions don't.

Your:

```text
coder blocked until scout produces scope
```

is the right approach.

The AGENTS.md philosophy is solid.

---

# Biggest Concern

The system is becoming a mini-agent platform inside Pi.

You are rebuilding:

* planning
* delegation
* activity tracking
* state machines
* progress UI
* overlays
* scope management

inside Pi.

That creates a maintenance burden.

The architecture is trending toward:

```text
Pi
 └─ Orchestrator
     └─ Subagent framework
         └─ Activity framework
             └─ Overlay framework
```

rather than:

```text
Pi
 └─ Lightweight orchestration extension
```

# Cache Conservation Audit

This system is actually quite cache-efficient.

Why?

Subagents receive:

```text
raw task string only
```

instead of:

```text
entire conversation
```

The trace explicitly shows:

```text
NO conversation history
NO project context
NO orchestration history
```

which is very good.

---

## Good decision

```ts
noContextFiles: true
```

for scouts.

Huge token savings.

---

## Bad decision

The activity feed stores:

```ts
rawText
MAX_RAW_TEXT = 10_000
```

per subagent.

For long investigations this accumulates quickly.

I'd reduce:

```ts
MAX_RAW_TEXT
```

from

```text
10000
```

to

```text
3000-5000
```

and retain only parsed state.

# Scope Enforcement Audit

Excellent concept.

However:

```text

```

is too rigid.

Refactors will suffer.

I'd switch to:

```text
allowed directories
```

plus

```text
max files
```

Example:

```json
{
  "directories": [
    "src/auth",
    "src/middleware"
  ],
  "maxFiles": 8
}
```

Much more robust.

# State Machine Audit

The external review in your docs is correct.

This is the largest engineering risk.

You have:

```text
text_delta
tool_start
tool_end
timer_tick
spinner_update
```

all touching feed state.

That's classic race-condition territory.

I would move to:

```ts
dispatch(action)
reducer(state, action)
```

100%.

No direct mutations.

# Final Verdict

The architecture is stronger than most Pi orchestrators I've seen because it enforces behavior through tools rather than prompts. The biggest opportunity is not adding more capabilities, but simplifying:

* remove dependence on `bash` for scouts
* make `edit` the only file-modification path
* reduce activity-feed complexity
* drop or de-emphasize the overlay
* move state management to a reducer model
* test rendering/state transitions rather than ANSI output

If the goal is **maximum Pi-native behavior, cache efficiency, and long-term maintainability**, I'd estimate roughly **20–30% of the orchestration/UI code could be removed while improving reliability and user experience.**
