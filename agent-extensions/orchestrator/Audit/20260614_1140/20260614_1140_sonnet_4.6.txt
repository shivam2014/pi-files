## Pi SDK Compliance Audit

### ✅ Correct API Usage

**Tool registration** (`delegate-tool.ts`, `plan-tool.ts`) — `pi.registerTool()` with `parameters: Type.Object(...)` from typebox is correct. `renderCall` / `renderResult` signatures match the SDK.

**Event hooks** (`index.ts`) — `pi.on("before_agent_start", ...)` and `pi.on("tool_call", ...)` are correct SDK hooks. The return value `{ systemPrompt: ... }` from `before_agent_start` is correct.

**Tool blocking** (`index.ts:148-151`) — `return { block: true, reason }` from the `tool_call` handler is the correct SDK pattern.

**Session creation** (`subagent-runner.ts`) — `createAgentSession({ cwd, model, tools, resourceLoader, sessionManager, authStorage, modelRegistry })` matches the SDK. `SessionManager.inMemory(cwd)` is correct for ephemeral subagents.

**`DefaultResourceLoader` with `systemPromptOverride`** — correct. `noContextFiles: true` suppresses `AGENTS.md` loading, which is intentional for subagents.

**Widget** (`plan-panel.ts`) — `ctx.ui.setWidget("orchestrator-status", lines)` / `setWidget(key, undefined)` to clear — correct SDK pattern.

**Command registration** (`commands.ts`) — `pi.registerCommand(name, { description, handler })` — correct.

**Shortcut registration** (`index.ts`) — `pi.registerShortcut("ctrl+q", { description, handler })` — correct.

**`pi.sendUserMessage(task, { deliverAs: "followUp" })`** (`commands.ts`) — correct SDK usage.

### ⚠️ Compliance Issues

**1. `pi.setActiveTools(["plan", "delegate"])` — unverified**

```typescript
pi.setActiveTools(["plan", "delegate"]);
```

This is called in `before_agent_start`. The method is not in the public `ExtensionAPI` type definition in `@earendil-works/pi-coding-agent`. It may exist as an undocumented/internal method. If it doesn't exist at runtime, the tool blocking falls back entirely to the `tool_call` handler safety net — which does work — but the intent (hiding tools from the model's tool list) wouldn't be achieved. **Verify this exists in types.** The audit doc already flags this.

**2. `ctx.ui.custom()` overlay — experimental**

```typescript
ctx.ui.custom(
    (tui, _theme, _keybindings, done) => { ... },
    { overlay: true, overlayOptions: { anchor: "right-center", width: "50%", maxHeight: "80%" } }
)
```

Marked experimental in pi docs. The `.catch()` fallback is good practice. But the `onHandle` callback pattern:

```typescript
onHandle: (handle: OverlayHandle) => { _peekHandle = handle; }
```

...is not documented in the SDK. If this API changes, `isPeekOpen()` silently returns `false` forever and peek becomes a no-op without error. Needs a try-catch around the whole `showPeek` call.

**3. Local interface re-declaration in `peek-overlay.ts`**

```typescript
interface Component {
    render(width: number): string[];
    handleInput?(data: string): void;
    invalidate(): void;
}
```

The audit doc already called this out. At runtime these are duck-typed, so it works — until pi adds a required method to `Component`. The fix is straightforward: import from `@earendil-works/pi-tui`. The tsconfig already maps it.

**4. Widget ID namespace collision risk**

```typescript
const WIDGET_KEY = "orchestrator-status";
```

Pi has no widget namespacing mechanism in the current SDK. If another extension uses the same key, they'll overwrite each other. Since this is a personal extension, the risk is low, but the correct pattern would be a UUID-prefixed key or reverse-domain convention: `"earendil/orchestrator-status"`.

**5. `require()` in `plan-panel.ts`**

```typescript
const fs = require('fs');
const path = require('path');
```

Pi extensions run under jiti/tsx which supports ESM. Using `require()` for `node:fs` in what is otherwise an ESM module is inconsistent and can cause issues under strict ESM. Use `import { existsSync, mkdirSync, writeFileSync } from "node:fs"` at top-level or dynamic `await import("node:fs")`.

**6. `require()` in `delegate-tool.ts` renderCall/renderResult**

```typescript
const comp = context.lastComponent ?? new (require("@earendil-works/pi-tui").Text)("", 0, 0);
```

Dynamic `require()` inside render callbacks is called on every render frame. This is a performance issue — the module is cached by Node after the first call, but the property lookup chain still runs every frame. More importantly, `renderCall` in `plan-tool.ts` does this correctly:

```typescript
import { Text } from "@earendil-works/pi-tui";
// ...
renderCall(args, theme, context) {
    return new Text(`⠋ Plan: ...`, 0, 0);
}
```

`delegate-tool.ts` should follow the same pattern. The inconsistency suggests `plan-tool.ts` was written after `delegate-tool.ts` and the pattern wasn't backported.

**7. `_batchLoadSubagent` as a mutable export**

```typescript
export let _batchLoadSubagent = 0;
```

This is a module-level counter exported mutably. Under hot reload (jiti), the module may re-execute, resetting the counter to 0 while subagents are in flight. The `globalThis` registry trick used in `plan-panel.ts` for timers should be applied here too. The `_reg()` pattern already in plan-panel is the right model.

**8. `subagent-runner.ts` — Peek not wired (P0 bug confirmed)**

The audit doc correctly identifies this. `registerPeekFeed()` is called, but `updatePeek()` for text deltas is called — wait, actually looking more carefully at line:

```typescript
updatePeek(event.assistantMessageEvent.delta);
```

It IS being called in `subagent-runner.ts`. But `updatePeek` in `peek-overlay.ts` has this guard:

```typescript
export function updatePeek(text: string): void {
    if (!_peekHandle || _peekHandle.isHidden()) return;
```

The peek only receives content if it's already open (`_peekHandle` exists). Since `showPeek` is only called from the `Ctrl+Q` shortcut handler — which creates `_peekHandle` on demand — content streaming works once the user opens it. But if the user presses `Ctrl+Q` after the subagent starts, the initial content (before the press) is lost. **The buffer should pre-accumulate lines even when peek is closed**, and replay them on open. This is the actual disconnect, not the missing `updatePeek` call.

## State Machine Issues (Coding Review)

**Race condition in `subagent-runner.ts`**

```typescript
// tool_execution_update handler
lastSub.outputPreview = preview; // direct mutation
```

The `tool_execution_update` handler mutates `feed.steps[x].substeps[y].outputPreview` directly while the 80ms render timer may be reading `feed` concurrently. JavaScript is single-threaded so there's no true data race, but the mutation happens outside the immutable reducer pattern used everywhere else, creating inconsistency. The `activity-feed.ts` exported functions all return new state — but `subagent-runner.ts` sometimes bypasses them.

**`addStep` in `test-unit.test.ts` vs `activity-feed.ts` — divergence**

The unit tests inline their own copy of `addStep` with direct mutation:
```typescript
function addStep(state: ActivityFeedState, label: string): void {
    state.steps.push(...) // mutates
```

But `activity-feed.ts` has been refactored to return new state:
```typescript
export function addStep(state: ActivityFeedState, label: string): ActivityFeedState {
    return { ...state, steps: [...state.steps, newStep] }
```

The tests are testing the old mutating version, not the current implementation. `npx vitest run` will pass but is testing dead code. This is a meaningful gap — the tests give false confidence.

**`completeCurrentStep` in activity-feed.ts vs test-unit.test.ts**

Same problem. Test inlines the old mutating version. The real implementation returns new state and the tests don't cover it.

## UX/UI Review

### What's working well

**The three-layer hierarchy is conceptually sound.** Plan panel (widget, persistent) → inline tool result progress (chat history) → peek overlay (on-demand) maps well to the information needs at each zoom level. This is better design than Claude Code's approach where everything competes in a single stream.

**Spinner centralization** (`spinner-state.ts`) is clean. One source of truth for frames, shared across all layers. The `advanceSpinner()` / `getSpinnerIndex()` / `resetSpinner()` split is correct — advance on render, reset on step transition.

**`compressOutput`** (ANSI strip + blank collapse) before returning to orchestrator is the right call. Subagent output often contains terminal decoration that would balloon context.

**`toolCallToSubstep`** label mapping is good UX — converts raw tool calls to human-readable substep descriptions. `"read" → "Reading src/auth.ts"` is exactly the right abstraction level for the plan panel.

**Caveman mode system prompt** is a solid token conservation pattern, consistently applied across all specialists.

### Issues

**1. Progress dots format mismatch between layers**

`SPEC-UI.md` specifies:
```
●○○ 1/3
```

But `plan-panel.ts` renders:
```typescript
lines.push(`Plan: ◆ ${goalTrunc}${prog ? `  ${prog}  ` : '  '}${elapsedStr}`);
// prog = `● ${completedCount}/${total}` (no individual dots)
```

The plan panel renders `● 2/4` (filled dot + count) while the spec calls for `●○○ 2/4` (one dot per step). The chat-level `renderActivityFeed` does render individual dots per step. **The two layers are visually inconsistent.** Plan panel loses the per-step granularity the spec requires.

**2. Substep collapse vs erase**

`VISION.md` and `SPEC-UI.md` both specify "collapse not erase" — completed substeps stay visible under their parent step. But `completePlanStep` in `plan-panel.ts` does:

```typescript
step.detailLines = undefined; // ← erases substep detail
```

Then saves only the last 3 as `substepLines`:
```typescript
(step as any).substepLines = step.detailLines.slice(-3);
```

So under the plan panel, a completed step shows at most 3 substeps, and they get dropped on the next `_renderWidget` call anyway due to the BUDGET=9 constraint. This is a deliberate tradeoff (token budget), but it violates the spec's stated invariant and should be documented as such rather than left as a silent delta from the vision doc.

**3. Tool detail indentation**

`SPEC-UI.md` specifies 8-space indent for tool detail:
```
        ⠋ Running: cat src/auth/validate.ts
```

`renderActivityFeed` in `activity-feed.ts` renders:
```typescript
lines.push(`        ${spinner} ${sub.toolDetail}`);
```

That's 8 spaces — correct. But `toolCallToSubstep` doesn't prefix with "Running:" for all cases:
```typescript
case "read":
    return `Reading ${normalizePath(...)}`;  // no "Running:" prefix
case "bash":
    return `Running: ${cmd.slice(0, 100)}`;  // has prefix
```

The spec says tool detail format is `Running: <command summary>` but the implementation only uses that prefix for bash. For read/grep/find/edit/write it uses verb phrases like "Reading", "Searching", "Finding". This is actually better UX than the spec — the spec's "Running:" prefix is generic and less informative. The spec should be updated to match the implementation rather than the reverse.

**4. `renderCombinedProgress` vs `renderActivityFeed` duplication**

Both functions in `activity-feed.ts` contain nearly identical error rendering logic (~40 lines each). The error path is duplicated verbatim. Extract to `renderErrorState(state, lines)` helper. As it stands, any bug fix in one must be manually mirrored in the other.

**5. Peek overlay missing content replay**

Already noted above. When user presses `Ctrl+Q` mid-subagent, they see only new content from that point forward. Pre-open content is lost. Fix: accumulate `_peekLines` unconditionally in `updatePeek` (remove the `_peekHandle` guard), and only skip the `_peekTui?.requestRender()` call. Content is buffered, replay happens when peek opens.

**6. `_orchestratorActivity` in `delegate-tool.ts` is module-level**

```typescript
let _orchestratorActivity: ReturnType<typeof createOrchestratorActivity> | null = null;
```

If two delegations run concurrently (which the `incrementDelegationCount` guard suggests is possible), this variable gets overwritten by the second delegation. The activity object for the first delegation is lost. Either enforce strict serialization (one delegation at a time) or make activity objects local to the execute closure.

**7. Scope extraction now redundant**

`extractFindingsFromOutput` and `extractAuditFromOutput` parse regex patterns from subagent text. But the scope for coder now comes from the structured `scope` parameter on the `delegate()` call, not from text parsing. The `_cachedScope` from text parsing in an older version has been removed in favor of `params.scope`. But `extractFindingsFromOutput` still does regex extraction, which is fine for findings/audit — just double-check `_cachedScope = params.scope` is always set before `runSubagent` is called when `params.specialist === "coder"`.

## Refactoring Priority (my view, differs slightly from AUDIT-REVIEW-2026-06-13.md)

**P0 — Break production:**
- Peek content replay (open peek mid-task → empty overlay)
- `_orchestratorActivity` concurrent write (second delegation clobbers first)
- Unit tests testing inlined dead code, not actual exports

**P1 — Spec violations:**
- Progress dots format in plan panel (`● 2/4` vs spec `●○○ 2/4`)
- `renderCombinedProgress` / `renderActivityFeed` error path duplication
- `require()` in render callbacks → move to top-level import

**P2 — Hygiene:**
- `pi.setActiveTools()` — verify exists in SDK types or document as monkey-patch
- `_batchLoadSubagent` — apply globalThis guard against hot-reload reset
- Widget ID namespacing
- `peek-overlay.ts` — import real types from `@earendil-works/pi-tui`

The AUDIT-REVIEW-2026-06-13.md's grading of B+/C/B/D+ is fair. Architecture is genuinely good — the refactor from a 1663-line monolith to 11 focused files with clear ownership is the right call. The testability gap (D+) is the real blocker for junior dev productivity, and the unit test divergence from actual implementation makes it actively misleading.
